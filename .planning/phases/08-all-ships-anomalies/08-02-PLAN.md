<task type="auto">
  <name>Task 1: Add shipType filter support to GET /api/anomalies</name>
  <files>src/app/api/anomalies/route.ts</files>
  <action>
    Extend the existing GET handler in `src/app/api/anomalies/route.ts` to support an optional `?shipType=` query param with values `tanker`, `cargo`, or `other` (absent = all).

    When `shipType` is provided, LEFT JOIN with the `vessels` table on `imo` and add a WHERE clause:
    - `tanker`: `AND v.ship_type BETWEEN 80 AND 89`
    - `cargo`: `AND v.ship_type BETWEEN 70 AND 79`
    - `other`: `AND (v.ship_type IS NULL OR v.ship_type &lt; 70 OR v.ship_type > 89)`
    - absent/`all`: no additional filter

    The JOIN must be a LEFT JOIN so anomalies for vessels not yet in the vessels table still appear when `shipType` is absent. Return the existing columns unchanged — do NOT add shipType to the response shape (the filter is server-side).

    Example final query structure when shipType=tanker:
    ```sql
    SELECT va.id, va.imo, va.anomaly_type as "anomalyType", va.confidence,
           va.detected_at as "detectedAt", va.resolved_at as "resolvedAt", va.details
    FROM vessel_anomalies va
    LEFT JOIN vessels v ON v.imo = va.imo
    WHERE va.resolved_at IS NULL
      AND v.ship_type BETWEEN 80 AND 89
    ORDER BY va.detected_at DESC
    ```

    Use parameterized query for the imo param (already present). The shipType filter is injected as a SQL string fragment (safe — it comes from a controlled switch/if on the query param value, not raw user input).
  </action>
