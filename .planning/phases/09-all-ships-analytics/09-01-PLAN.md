    In src/lib/db/analytics.ts, update getTrafficByChokepoint signature to accept optional third param `shipTypeFilter: ShipTypeFilter = 'all'`. Build the SQL WHERE clause for ship type via a controlled switch (never inject raw user input):

    ```typescript
    const shipTypeClause = (() => {
      switch (shipTypeFilter) {
        case 'tanker': return 'AND v.ship_type BETWEEN 80 AND 89';
        case 'cargo':  return 'AND v.ship_type BETWEEN 70 AND 79';
        case 'other':  return 'AND (v.ship_type IS NULL OR v.ship_type NOT BETWEEN 70 AND 89)';
        default:       return ''; // 'all' — no filter
      }
    })();
    ```

    Inject shipTypeClause into the SQL query's WHERE block (after the spatial bounds conditions).

    For the returned vesselCount: when a ship type filter is active, COUNT(DISTINCT vp.mmsi) already returns the filtered set because the WHERE clause limits which rows are aggregated. Keep the tankerCount column in the query but it will reflect the filtered tanker count (0 or same as vesselCount for tanker filter). This preserves backward compatibility with TrafficChart which reads both fields.

    Write/update tests in src/lib/db/analytics.test.ts to cover: filter='all' returns no ship_type clause in generated SQL (or mock DB call returns expected shape), filter='tanker' injects correct clause. Use existing test patterns from the file.
